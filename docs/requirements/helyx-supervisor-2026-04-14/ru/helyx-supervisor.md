# PRD: Helyx Supervisor — Супервизор сессий

## 1. Обзор

Helyx Supervisor — это постоянно работающий процесс-наблюдатель, который следит за здоровьем всех компонентов бота Helyx: сессий Claude Code, очередей сообщений, голосовых статусов и Docker-контейнеров. При обнаружении аномалий супервизор автоматически выполняет восстановление, объясняет ситуацию через LLM (gemma4:e4b / Ollama) и уведомляет пользователя в отдельный Telegram-топик. Любое сообщение в supervisor topic вызывает интерактивный ответ с оценкой состояния системы.

Вдохновлён архитектурой OpenClaw Gateway: центральный управляющий контур с retry policy, session health tracking и multi-agent routing.

---

## 2. Контекст

- **Продукт:** Helyx — Telegram-бот, управляющий сессиями Claude Code
- **Модуль:** scripts/admin-daemon.ts (расширяется) + новый supervisor.ts
- **Пользователь:** Один администратор (altsay)
- **Стек:** TypeScript/Bun, PostgreSQL, Docker, tmux, Ollama (gemma4:e4b)
- **Режимы деплоя:** Docker (bot + postgres) ИЛИ хост-процесс (bun main.ts)

---

## 3. Проблема

1. Сессии зависают без уведомления (channel subprocess умер, но запись в DB осталась активной)
2. `message_queue` накапливает необработанные сообщения при сбое poller'а
3. `active_status_messages` остаются "зомби" в Telegram после краша бота
4. `voice_status_messages` остаются "downloading..." навсегда
5. Единственное восстановление — полный рестарт Docker (ручной)
6. Telegram TTS 429 — без retry, голосовые теряются
7. Нет единой точки наблюдения за состоянием всех сессий

---

## 4. Цели

- Автоматически обнаруживать и восстанавливать зависшие состояния без ручного вмешательства
- Перезапускать отдельные channel subprocess без полного рестарта Docker
- Использовать LLM (qwen3:8b) для человекочитаемой диагностики
- Уведомлять в отдельный Telegram-топик о любых инцидентах
- Работать в обоих режимах деплоя (Docker + хост)

---

## 5. Не-цели

- Замена существующего `admin-daemon.ts` (расширение, не замена)
- Мониторинг внешних сервисов (Groq, Anthropic API, Yandex) — только локальные компоненты
- Автоматический рестарт Docker при падении бота (требует подтверждения)
- Веб-UI или отдельный дашборд (используется существующий `/monitor`)

---

## 6. Функциональные требования

**FR-1: Мониторинг сессий**
Каждые 60 секунд проверять `sessions` + `active_status_messages`:
- Сессия активна, но heartbeat в `active_status_messages.updated_at` не обновлялся >2 мин → сессия "зависла"
- Сессия активна, но в `message_queue` есть необработанные сообщения >5 мин → очередь "застряла"

**FR-2: Мониторинг очереди**
Каждые 60 секунд проверять `message_queue`:
- Записи с `delivered = false` старше 5 минут → алерт с inline-кнопками (🔄 Перезапустить / ✅ Игнорировать)
- Авто-рестарт убран — решение принимает пользователь через кнопку

**FR-3: Восстановление голосовых статусов**
Каждые 5 минут проверять `voice_status_messages`:
- Записи старше 3 минут → редактировать Telegram-сообщение + удалять из DB

**FR-4: Перезапуск channel subprocess + верификация восстановления**
При обнаружении зависшей сессии:
1. Отправить алерт в supervisor topic
2. Вставить `admin_commands(command='proj_start', payload={path})` → admin-daemon выполнит
3. Опрашивать `active_status_messages` каждые 5 секунд в течение 60 секунд
4. Если heartbeat возобновился → отправить ✅ восстановлено
5. Если нет — отправить ⛔ с кнопкой 🔄 повторить

**FR-5: LLM-диагностика (gemma4:e4b via Ollama)**
При каждом инциденте и запросе статуса:
1. Вызвать Ollama `/api/chat` с параметром `think: false`, модель `gemma4:e4b`
2. Системный промпт ограничивает модель контекстом мониторинга Helyx
3. Включить комментарий LLM в уведомление (best-effort, таймаут 10с)

**FR-6: Telegram-уведомления**
Все алерты отправлять в отдельный топик (SUPERVISOR_TOPIC_ID):
- Инцидент: тип + проект + elapsed + LLM-комментарий
- Recovery: действие + результат (✅ или ⛔)
- Статус-рассылка: каждые 5 минут — сессии, очередь, здоровье процессов + оценка LLM
- Алерты о застрявшей очереди содержат inline-кнопки
- Все Telegram-вызовы — через `tgPost()` с retry при 429

**FR-7: Интеграция с admin-daemon**
Супервизор работает внутри `admin-daemon.ts` как дополнительный setInterval-цикл, используя уже открытое DB-соединение и `runShell`/`runCommand` функции.

**FR-8: Чат в supervisor topic**
Любое сообщение в supervisor topic вызывает немедленный ответ со статусом:
- Живые данные из DB: сессии, очередь, здоровье процессов
- LLM-оценка текущего состояния с учётом вопроса пользователя
- Модель отвечает по-русски, в рамках контекста мониторинга Helyx

---

## 7. Нефункциональные требования

**NFR-1:** Супервизор не должен потреблять >50MB RAM и <2% CPU в idle
**NFR-2:** LLM-диагностика (gemma4:e4b) — таймаут 10 секунд, не блокирует recovery
**NFR-3:** При недоступности Ollama — пропустить LLM, отправить алерт без объяснения
**NFR-4:** Все действия логировать в `process_health` таблицу с типом 'supervisor'
**NFR-5:** Алерты идемпотентны — не дублировать уведомления по одному инциденту

---

## 8. Ограничения

- Работает только с компонентами, которые имеют запись в DB (сессии без DB-tracking не видны)
- gemma4:e4b должен быть доступен в Ollama (уже установлен)
- `SUPERVISOR_CHAT_ID` и `SUPERVISOR_TOPIC_ID` должны быть заданы в .env
- Перезапуск Docker контейнера требует sudo или docker группы (уже настроено)
- admin-daemon должен быть запущен для выполнения tmux команд

---

## 9. Граничные случаи

- **Ollama недоступна:** отправить алерт без LLM-объяснения, продолжить работу
- **Telegram API 429:** retry через retry_after секунд (та же логика что в TTS)
- **admin-daemon не запущен:** залогировать ошибку, не пытаться писать в admin_commands
- **Инцидент длится >30 мин без восстановления:** эскалация алерта с пометкой "требует ручного вмешательства"
- **Рестарт супервизора:** не дублировать уже закрытые инциденты (проверять по timestamp)

## 10. Критерии приёмки (Gherkin)

```gherkin
Feature: Supervisor — автоматическое восстановление зависших сессий

  Scenario: Зависший channel subprocess
    Given в active_status_messages есть запись с updated_at > 2 минуты назад
    When supervisor-цикл срабатывает
    Then supervisor вставляет proj_start в admin_commands
    And admin-daemon перезапускает tmux-окно
    And в Telegram-топик отправляется алерт с LLM-объяснением
    And запись удаляется из active_status_messages

  Scenario: Застрявшая очередь
    Given в message_queue есть записи status='pending' старше 5 минут
    When supervisor-цикл срабатывает
    Then supervisor определяет project по session_id
    And вставляет proj_start в admin_commands
    And отправляет алерт в Telegram-топик

  Scenario: Зависший голосовой статус
    Given в voice_status_messages есть запись старше 3 минут
    When supervisor проверяет voice статусы
    Then редактирует Telegram-сообщение на "⚠️ Бот перезапущен — голосовое не обработано"
    And удаляет запись из voice_status_messages

  Scenario: Ollama недоступна
    Given Ollama не отвечает или возвращает ошибку
    When supervisor пытается получить LLM-диагностику
    Then supervisor пропускает LLM и отправляет алерт без объяснения
    And продолжает работу нормально

  Scenario: Инцидент длится >30 минут
    Given supervisor обнаружил инцидент более 30 минут назад
    And восстановление не помогло
    When supervisor-цикл срабатывает снова
    Then отправляется эскалационный алерт "требует ручного вмешательства"
```

## 11. Верификация

- **Где тестировать:** локально, имитируя зависание через прямую UPDATE в DB
- **Как тестировать:**
  - `UPDATE active_status_messages SET updated_at = now() - interval '5 minutes' WHERE key = 'test'`
  - Проверить что через 60 сек приходит Telegram-алерт с LLM-объяснением
  - Проверить что tmux-окно перезапустилось
- **Observability:**
  - `process_health` таблица: запись 'supervisor' с uptime и счётчиком инцидентов
  - `/monitor` команда показывает статус супервизора
  - `/tmp/admin-daemon.log` содержит логи всех действий супервизора
