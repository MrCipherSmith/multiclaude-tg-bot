# ПРД: Команды Telegram-бота для скилов, команд и хуков Claude Code

## 1. Обзор

Добавить в Telegram-бот команды `/skills`, `/commands`, `/hooks`, которые динамически читают доступные инструменты с хоста через HTTP-эндпоинт и отображают их смешанным UI (заголовок + inline-кнопки с описаниями). При выборе инструмента без аргументов — бот запрашивает уточнение, затем отправляет команду в активную Claude-сессию.

---

## 2. Контекст

- **Продукт:** @GoodeaAIBot (Telegram-бот)
- **Модуль:** `bot/commands/`, `bot/handlers.ts`, `main.ts` (HTTP-эндпоинт)
- **Роль пользователя:** Владелец бота (одиночный пользователь)
- **Стек:** Bun + TypeScript, grammY, PostgreSQL, Docker
- **Хост:** Скилы — `~/.claude/skills/`, команды — `~/.claude/commands/`, хуки — `~/.claude/settings.json`
- **Бот API:** `http://localhost:3847`

---

## 3. Проблема

Пользователь накопил десятки скилов, команд и хуков в глобальном Claude Code, но не может быстро их использовать из Telegram — приходится помнить названия наизусть или смотреть в файлы на сервере.

---

## 4. Цели

- Предоставить `/skills`, `/commands`, `/hooks` как точки входа из Telegram
- Показывать актуальный список инструментов динамически (без рестарта бота)
- Показывать краткое описание из frontmatter рядом с кнопкой
- Поддерживать интерактивный ввод аргументов перед инвокацией
- Отправлять выбранную команду в активную Claude-сессию как пользовательский промпт

---

## 5. Non-Goals

- Управление скилами/командами (создание, удаление, редактирование)
- Синхронизация между несколькими пользователями
- Поддержка нескольких профилей Claude (`~/.claude`)
- Пагинация (если скилов > 50 — вне скопа)
- История вызовов инструментов

---

## 6. Функциональные требования

**FR-1: Команды бота**
- `/skills` — показывает список скилов из `~/.claude/skills/`
- `/commands` — показывает список команд из `~/.claude/commands/`
- `/hooks` — показывает список хуков из `~/.claude/settings.json` (секция `hooks`)

**FR-2: HTTP-эндпоинт на хосте**
- `GET /api/tools/skills` — возвращает `[{ name, description, requiresArgs }]`
- `GET /api/tools/commands` — возвращает `[{ name, description, requiresArgs }]`
- `GET /api/tools/hooks` — возвращает `[{ event, matcher, command }]`
- Парсит frontmatter из `.md` / `SKILL.md` файлов
- Определяет `requiresArgs`: если в файле есть секция `## Arguments` или `args:` в frontmatter — `true`; иначе fallback по hardcoded-списку без аргументов (`commit`, `push`, `clean-gone`, `hookify-list`, `hookify-configure`)

**FR-3: UI — смешанный**
- Сообщение = заголовок категории + inline-кнопки
- Каждая кнопка: `<name> — <description (≤60 chars)>`
- Кнопки разбиты по строкам: 1 кнопка на строку (для читаемости описаний)

**FR-4: Инвокация без аргументов**
- Если `requiresArgs = false` → бот немедленно отправляет `/<name>` в активную Claude-сессию через `message_queue`

**FR-5: Инвокация с аргументами**
- Если `requiresArgs = true` → бот отвечает: `Введи аргументы для /<name>:`
- Следующее сообщение пользователя воспринимается как аргументы
- Бот отправляет `/<name> <аргументы>` в активную сессию
- Таймаут ожидания аргументов: 5 минут (после — сброс)

**FR-6: Динамическое чтение**
- При каждом вызове `/skills`, `/commands`, `/hooks` бот запрашивает актуальный список с эндпоинта (не кеш)

---

## 7. Нефункциональные требования

**NFR-1:** Эндпоинт отвечает за < 500 мс  
**NFR-2:** Если эндпоинт недоступен — бот отвечает понятной ошибкой, не падает  
**NFR-3:** Inline-кнопки не превышают лимит Telegram (64 байта на `callback_data`)  
**NFR-4:** Состояние "ожидание аргументов" хранится в памяти (Map), не в БД  

---

## 8. Ограничения

- Бот работает в Docker, хост доступен через `host.docker.internal` или прямым портом
- HTTP-эндпоинт добавляется в существующий `main.ts` (Hono/Bun)
- Парсинг frontmatter — библиотека `gray-matter` или ручной regex (уже есть в проекте?)
- `callback_data` для кнопок: `skill:<name>` / `cmd:<name>` / `hook:<index>` (≤ 64 байта)

---

## 9. Edge Cases

- Скил/команда с именем длиннее 30 символов → обрезать в кнопке
- `requiresArgs` не определён (нет секции, нет в fallback-списке) → считать `true` (безопасный дефолт)
- Активная сессия отсутствует (standalone mode) → показать предупреждение перед инвокацией
- Пользователь нажал кнопку, но потом отправил другое сообщение до ввода аргументов → отменить ожидание, обработать новое сообщение как обычный промпт
- Хуков нет в `settings.json` → сообщение "Хуки не настроены"

## 10. Acceptance Criteria (Gherkin)

```gherkin
Feature: Telegram команды для инструментов Claude Code

  Scenario: Просмотр списка скилов
    Given пользователь авторизован в боте
    When пользователь отправляет /skills
    Then бот делает GET /api/tools/skills
    And отображает заголовок "⚡ Skills"
    And показывает inline-кнопки: одна кнопка на строку
    And каждая кнопка содержит название и описание ≤ 60 символов

  Scenario: Инвокация скила без аргументов
    Given список скилов отображён
    When пользователь нажимает кнопку "commit — Create a git commit"
    And requiresArgs = false для "commit"
    Then бот отправляет "/commit" в активную Claude-сессию через message_queue

  Scenario: Инвокация скила с аргументами
    Given список скилов отображён
    When пользователь нажимает кнопку "hookify — Creates Claude Code hooks..."
    And requiresArgs = true для "hookify"
    Then бот отвечает "Введи аргументы для /hookify:"
    When пользователь отправляет "запрети коммиты без тестов"
    Then бот отправляет "/hookify запрети коммиты без тестов" в активную сессию

  Scenario: Эндпоинт недоступен
    Given сервис на хосте не отвечает
    When пользователь отправляет /skills
    Then бот отвечает "Не удалось загрузить список: сервис недоступен"
    And не падает с необработанной ошибкой

  Scenario: Таймаут ожидания аргументов
    Given бот ожидает аргументы для /hookify
    When проходит 5 минут без ответа пользователя
    Then бот отменяет ожидание
    And следующее сообщение обрабатывается как обычный промпт
```

---

## 11. Верификация

- **Ручное тестирование:** отправить `/skills`, `/commands`, `/hooks` в Telegram, проверить кнопки и описания
- **Тест инвокации:** нажать кнопку безаргументного скила → убедиться в `message_queue` через psql
- **Тест уточнения:** нажать кнопку hookify → ввести текст → проверить `message_queue`
- **Тест ошибки:** остановить bot API → вызвать `/skills` → убедиться в корректном сообщении об ошибке
- **Observability:** логи в `bot/` при инвокации через кнопку — `appendLog(sessionId, chatId, "tools", "invoked /hookify")`
